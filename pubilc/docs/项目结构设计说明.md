# ELEVEN Blog Tuner - 项目结构设计说明

## 1. 项目概述

**项目名称**: ELEVEN Blog Tuner\
**项目目的**: 个人博客 AI 智能助手，根据个人笔记学习并生成相似文笔的文章\
**技术栈**: Python + FastAPI + Vue3\
**核心功能**: 风格学习、文章生成、文章发布

***

## 2. 架构设计原则

### 2.1 分层架构

项目采用清晰的分层架构，确保职责分离：

- **API 层**: 处理 HTTP 请求/响应
- **Gateway 层**: 网关层 - 控制系统核心
- **Core 层**: 核心业务逻辑和配置
- **Agent 层**: AI Agent 实现
- **RAG 层**: 检索增强生成
- **LLM 层**: 大语言模型接口
- **Tools 层**: 工具集管理

### 2.2 模块化设计

- 每个模块独立负责特定功能
- 模块间通过接口通信，降低耦合
- 便于单元测试和独立部署

### 2.3 可扩展性

- 支持多 LLM 提供商（OpenAI、Claude、本地模型）
- 插件化工具注册机制
- 可插拔的 RAG 组件

### 2.4 异常处理体系

- **分层异常**：从基础异常到具体业务异常的层级结构
- **标准化错误响应**：统一的错误格式和状态码
- **详细错误信息**：包含错误码、错误信息和详细上下文
- **可追踪性**：支持错误链和详细日志

## 4. 核心流程说明

### 4.1 文章生成流程

```
用户输入
    ↓
Boss Agent (任务调度)
    ↓
System Agent (获取风格配置)
    ↓
Summary Agent (分析上下文)
    ↓
RAG Pipeline (检索相关知识)
    ↓
Writer Agent (撰写文章)
    ↓
Review Agent (质量审查)
    ↓
输出文章
```

### 4.2 RAG 处理流程

```
原始文档
    ↓
Document Washer (清洗)
    ↓
Chunker (分块)
    ↓
Embedding Service (向量化)
    ↓
Vector DB (存储)

查询时:
查询文本
    ↓
Embedding (向量化)
    ↓
Searcher (向量检索 Top-K*2)
    ↓
Reranker (精排 Top-K)
    ↓
返回结果
```

### 4.3 LLM 调用流程

```
用户请求
    ↓
Memory (加载历史)
    ↓
LLM Factory (创建对应 LLM)
    ↓
BaseLLM.chat/stream_chat
    ↓
Tools? (判断是否需要工具)
    ↓
    ├─ Yes → Function Calling → 执行工具 → 返回结果
    └─ No → 直接返回 LLM 输出
```

### 4.4 Gateway 层流程

```
用户请求
    ↓
API 层 (接收 HTTP 请求)
    ↓
Gateway 层 (控制系统核心)
    ↓
┌──────────────────────────────────────┐
│ 1. 任务创建与管理                     │
│ 2. 与 BossAgent 深度绑定             │
│ 3. 与 SystemAgent 深度绑定            │
│ 4. 24小时运行与自动处理任务链          │
└──────────────────────────────────────┘
    ↓
Boss Agent (任务调度)
    ↓
分析任务类型
    ↓
┌──────────────────────────────────────┐
│ 1. SystemAgent ← 获取风格配置         │
│ 2. SummaryAgent ← 分析上下文          │
│ 3. WriterAgent ← 撰写文章            │
│ 4. ReviewAgent ← 质量审查             │
└──────────────────────────────────────┘
    ↓
Gateway 层 (状态管理与结果返回)
    ↓
API 层 (返回响应给用户)
```

### 4.5 BaseAgent 架构

```
BaseAgent
├── LLM 集成 (call_llm)
├── 记忆系统 (add_to_memory, get_memory_history)
├── 工具系统 (add_tool, call_tool)
├── 子 Agent 调用 (call_agent)
└── 消息构建 (build_messages)
```

### 4.6 AgentProtocol 熔断机制

```
AgentProtocol
├── CircuitBreaker (熔断器)
│   ├── CLOSED (关闭状态 - 正常服务)
│   ├── OPEN (开启状态 - 熔断中)
│   └── HALF_OPEN (半开状态 - 试探性服务)
├── 失败阈值: 5次
├── 恢复超时: 30秒
└── 半开状态最大试探调用: 3次

执行流程:
1. 调用前检查熔断状态
2. 如果熔断开启，抛出 CircuitBreakerOpenError
3. 执行 Agent 调用
4. 成功: 记录成功，半开状态下累计成功达到阈值则关闭熔断
5. 失败: 记录失败，达到阈值则开启熔断
```

## 5. 配置管理

### 5.1 配置优先级

1. 环境变量 (最高优先级)
2. .env 文件
3. 默认值 (最低优先级)

### 5.2 配置分类

| 类别  | 配置项                                                                                | 说明    |
| --- | ---------------------------------------------------------------------------------- | ----- |
| API | API\_BASE\_URL, API\_KEY                                                           | 服务配置  |
| LLM | LLM\_PROVIDER, LLM\_MODEL, LLM\_TEMPERATURE                                        | 模型配置  |
| RAG | VECTOR\_DB\_PATH, EMBEDDING\_MODEL, CHUNK\_SIZE, RERANK\_MODEL, USE\_LOCAL\_RERANK | 检索配置  |
| DB  | DATABASE\_URL                                                                      | 数据库配置 |
| Log | LOG\_LEVEL                                                                         | 日志级别  |

## 7. 扩展点设计

### 7.1 添加新的 LLM 提供商

1. 在 `llm/` 下创建新的 provider 文件
2. 继承 `BaseLLM` 实现 `chat()` 和 `stream_chat()`
3. 在 `factory.py` 的 `_providers` 中注册

### 7.2 添加新的 Agent

1. 在 `agents/` 下创建新的 agent 文件
2. 继承 `BaseAgent` 实现 `execute()`
3. 在 `AgentProtocol._init_builtin_agents()` 中注册

### 7.3 添加新的 Tool

1. 在 Agent 的 `_init_tools()` 方法中使用 `add_tool()` 注册
2. 实现具体的工具逻辑
3. 通过 `call_tool()` 调用

## 8. 技术选型说明

| 组件        | 选型                    | 理由              |
| --------- | --------------------- | --------------- |
| Web 框架    | FastAPI               | 异步支持、自动文档、类型提示  |
| 配置管理      | pydantic-settings     | 类型安全、.env 支持、验证 |
| 向量数据库     | Chroma (已实现)          | 轻量、本地优先、易集成     |
| Embedding | sentence-transformers | 本地运行、无需 API     |
| 任务队列      | Celery (计划中)          | 异步处理、定时任务       |
| 缓存        | Redis (计划中)           | 高速缓存、消息队列       |

## 9. 开发规范

### 9.1 命名规范

- 包名: 小写 + 下划线 (`eleven_blog_tunner`)
- 类名: 大驼峰 (`BaseAgent`, `RAGPipeline`)
- 函数/方法: 小写 + 下划线 (`get_settings`, `process_document`)
- 常量: 大写 + 下划线 (`CHUNK_SIZE`, `MAX_TOKENS`)

### 9.2 代码风格

- 使用 Black 格式化 (line-length: 100)
- 使用 Ruff 进行代码检查
- 类型注解必须完整
- 文档字符串使用双引号

### 9.3 提交规范

```
feat: 新功能
fix: 修复 bug
docs: 文档更新
style: 代码格式调整
refactor: 重构
test: 测试相关
chore: 构建/工具相关
```
